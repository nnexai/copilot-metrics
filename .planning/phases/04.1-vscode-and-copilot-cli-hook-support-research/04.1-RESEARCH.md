# Phase 4.1: VS Code and Copilot CLI Hook Support Research

## Primary Sources

- VS Code Copilot hooks documentation: https://code.visualstudio.com/docs/copilot/customization/hooks
- GitHub Copilot CLI hooks reference: https://docs.github.com/en/copilot/reference/hooks-reference
- Local Copilot CLI help: `copilot help config` and `copilot --help` from the installed CLI on this machine.

## Findings

VS Code supports Copilot agent hooks in preview through JSON config. Supported VS Code hook events include `SessionStart`, `SessionEnd`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `Stop`, and `Error`.

GitHub's Copilot CLI hooks reference documents lower camel case hook events, while the VS Code hooks article states VS Code uses the same hook format as Claude Code and Copilot CLI for compatibility. This means the default shared `both` config should use the CLI-native lower camel case format. VS Code-only PascalCase config remains available for users who want the VS Code event names exactly.

The CLI help confirms hook definitions can be configured through Copilot config. Local repository hook files under `.github/hooks/*.json` remain the least-surprise local install target for this project.

## Design Implications

- Keep telemetry as two explicit sources: `vscode` and `copilot-cli`.
- Keep hook events as a third import stream, but store enough evidence to retain source/session labels and avoid treating all events as one logical task.
- Add `--surface both|vscode|copilot-cli` to hook preview/install.
- Default `--surface both` should emit the CLI-native lower camel case format because VS Code documents compatibility with the Copilot CLI hook format.
- Normalize hook payload field names from both schemas in the redacting logger.
- Hook commands should output JSON and should not rely only on non-standard hook `env` support; command strings should carry `COPILOT_METRICS_HOME`.

## Validation Notes

- Automated tests should verify both hook config formats and payload normalization.
- Manual Copilot CLI validation must not disable repository customization discovery when validating hooks.
- Manual Copilot CLI validation should use cheap mini models such as `gpt-5-mini`; the goal is telemetry and hook shape, not answer quality.
- VS Code hook validation should be documented/manual unless a reliable local non-interactive VS Code extension host harness is added later.
