# Phase 4.1 Summary: VS Code and Copilot CLI Hook Support

## Outcome

Inserted and executed a corrective research phase after discovering the hook setup model was too Copilot CLI-centric. VS Code Copilot hooks and Copilot CLI hooks are now represented explicitly and backed by primary documentation.

## Implemented

- Added `--surface both|vscode|copilot-cli` support for hook preview/install.
- Default `both` emits Copilot CLI-compatible lower camel case hook events; `vscode` emits VS Code PascalCase hook events.
- Generated hook files now match the documented `version` plus `hooks` shape and include `bash`, `command`, `env`, `timeout`, and `timeoutSec`.
- Hook commands run `hook-log --quiet` so they log local JSONL without sending unnecessary hook output back to Copilot.
- Hook logger normalizes VS Code-style payload fields such as `hook_event_name`, `session_id`, and `tool_name`.
- The local config now separates VS Code and Copilot CLI sources while sharing the hook JSONL ingestion path.
- README and release docs now document both surfaces and cheap-model manual validation.

## Research Basis

- VS Code hooks documentation confirms workspace hook files under `.github/hooks/*.json`, PascalCase event names, and compatibility with Copilot CLI/Claude-style hook format.
- GitHub Copilot hooks reference confirms Copilot CLI loads repository-level `.github/hooks/*.json`, user-level hooks, settings hooks, and plugin hooks, with lower camel case event names.
