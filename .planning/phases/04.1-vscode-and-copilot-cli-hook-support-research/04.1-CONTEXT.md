# Phase 4.1: VS Code and Copilot CLI Hook Support Research - Context

## Trigger

The Phase 4 release work treated hooks as effectively Copilot CLI-only. User correction: VS Code also has hook support, so this needs a separate research-backed phase before release readiness can be considered complete.

## User Intent

- Track VS Code Copilot and Copilot CLI as first-class sources.
- Support OpenTelemetry ingestion for both surfaces.
- Support hook setup for both surfaces where supported by the tools.
- Preserve label attribution evidence by source and session.
- Keep full prompt/content capture disabled by default.
- Avoid inferred or guessed hook schemas.

## Existing State

- `src/paths.js` resolves separate VS Code and Copilot CLI telemetry JSONL files.
- `src/otel.js` and `src/ingest.js` support `--source vscode`, `--source copilot-cli`, and `--source hooks`.
- `src/setup.js` generated a hook config, but the previous model did not make VS Code hook support explicit.
- `scripts/manual-copilot-cli-flow.js` validates Copilot CLI telemetry and should validate hook JSONL collection locally.

## Required Correction

The configuration model needs explicit surfaces:

- `vscode`: VS Code Copilot Chat OTel plus VS Code agent hooks.
- `copilot-cli`: Copilot CLI OTel plus Copilot CLI hooks.
- `both`: one default hook config shape that is documented as compatible with both when supported by the underlying tools.

## Non-Goals

- No dashboard.
- No official GitHub billing reconciliation.
- No prompt/content storage by default.
- No CI invocation of real Copilot CLI or VS Code.
